## Context

GlowFrame has no mechanism for users or support staff to verify the running build version without opening developer tools. The `package.json` `version` field exists and is the canonical version source. The `HelpDialog` (opened with `?`) is the natural home for this information — it is already a utility panel that power users visit. `vite.config.ts` already uses `defineConfig` and has a `resolve.alias` block; adding a `define` entry is additive and low-risk.

## Goals / Non-Goals

**Goals:**
- Inject `package.json` version into the build as a Vite `define` constant (`import.meta.env.VITE_APP_VERSION`).
- Render the version as `v<semver>` in the `HelpDialog` footer with muted, small typography.
- Add a TypeScript type declaration so `import.meta.env.VITE_APP_VERSION` is typed as `string`.
- Cover version rendering with a Vitest unit test.

**Non-Goals:**
- Runtime version fetching or network requests.
- Displaying version anywhere outside the `HelpDialog`.
- Version checking / update prompts.
- Changelog or release notes display.

## Decisions

### D-1: Vite `define` over `import.meta.env.VITE_*` env file

`VITE_*` env vars require a `.env` file committed to the repo or set in CI, making them fragile for automated builds. Using `define: { 'import.meta.env.VITE_APP_VERSION': JSON.stringify(process.env.npm_package_version) }` in `vite.config.ts` reads the value directly from the npm lifecycle environment variable, which is always populated when running `npm run build` / `npm run dev`. This is zero-config and fully consistent with the F-210 requirement spec.

### D-2: Type declaration in `src/vite-env.d.ts`

The existing `src/vite-env.d.ts` (generated by Vite) augments `ImportMetaEnv`. Adding `readonly VITE_APP_VERSION: string` there keeps types co-located with other env var declarations and avoids a separate `d.ts` file.

### D-3: Render in `HelpDialog` footer, not `DialogHeader`

The footer is the conventional placement for meta/administrative information (version, copyright) and keeps the primary content — the shortcut table — visually uncluttered. A `<p>` with Tailwind class `text-xs text-muted-foreground text-center mt-4` achieves the required de-emphasis without introducing new design tokens.

## Risks / Trade-offs

- **`npm_package_version` may be undefined outside npm scripts** (e.g., running `vite build` directly). Mitigation: wrap as `JSON.stringify(process.env.npm_package_version ?? 'unknown')` so the constant is always a valid string. Unit test mocks the constant directly, so this edge case does not affect tests.
- **`import.meta.env` is stubbed in Vitest's jsdom environment.** Mitigation: use `vi.stubEnv` or directly assign `import.meta.env.VITE_APP_VERSION` in the test setup, which Vitest supports natively.
